# Roadmap вҖ” Second Quantization (Track D, feeds into Track A)

See [notes/roadmap.md](../roadmap.md) for the status table and how this track fits into the overall
plan. This track replaces the old "QFT groundwork" placeholder in
[quantum-theory-foundations.md](quantum-theory-foundations.md#basic-quantum-field-theory-formalization)
with a concrete, ordered construction, algebraic first вҖ” no Hilbert-space completion, closable
operators, or self-adjointness/spectral theory until the algebraic Linked Cluster Theorem is done.

Lives in its own directory, `LeanCondensedMatter/SecondQuantization/`, under the `SecondQuantization`
namespace вҖ” deliberately separate from `QuantumTheory/`/`QuantumTheory` (Track A): second
quantization is its own construction (Fock space, creation/annihilation, CCR/CAR), not an
extension of the axiomatic single-particle postulates in `Postulates.lean` et al. See
`notes/conventions.md`'s "one directory per track" rule.

## Primary line: finite-mode fermions

**As of this redesign, the finite-mode *fermionic* case is the primary line toward the Linked
Cluster Theorem**, not bosons. The bosonic development (`Bosonic/Occupation.lean`) continues in
parallel but is no longer the critical path. Rationale: condensed-matter LCT applications (the
project's actual target вҖ” see `notes/model-and-assumptions.md`) are lattice-fermion models, and
finite modes sidestep the analytic issues (completion, unboundedness) this track is explicitly
deferring past the algebraic result.

`QuantumTheory.Statistics` (`Common/Statistics.lean`) вҖ” a `boson`/`fermion` enum with an exchange sign
`zetaInt : Statistics вҶ’ в„Ө` (`+1`/`-1`) вҖ” exists so both occupation-number representations, and a
future unified CCR/CAR statement (`aбөўaвұјвҖ  - О¶ aвұјвҖ aбөў = Оҙбөўвұј`), can refer to the same sign convention
without duplicating it. It does **not** unify the occupation-number *type* itself: bosonic
occupation (`Mode вҶ’вӮҖ в„•`, arbitrary multiplicity) and fermionic occupation (`Finset Mode`, `0`/`1`
only) are different types for a reason (Pauli exclusion), and forcing them into one representation
parameterized by `Statistics` would be more awkward than two separate, simple representations.

## Dependency order (fermionic primary line)

```
One-particle space (Mode, [DecidableEq Mode])
        вҶ“
Fermionic occupation-number basis (Finset Mode)
        вҶ“
Fermionic Fock space (algebraic, finite-particle)
        вҶ“
Fermionic creation / annihilation operators (with sign factors)
        вҶ“
Canonical anticommutation relations
        вҶ“
Hamiltonians (number operator, free, interaction)
        вҶ“
Gibbs state / partition function / free energy (reuse Track A/C infrastructure)
        вҶ“
Dyson/Duhamel expansion
        вҶ“
Linked Cluster Theorem (reuse Track B's mu_eq_prod_restrict)
```

The bosonic line (`Bosonic/Occupation.lean` onward) mirrors this shape one step behind, without the
sign factors: see "Bosonic line" below.

Field operators `ПҶ(x)` are NOT the starting point in either line вҖ” they are introduced only after
creation and annihilation operators exist, as a later derived construction.

## Common statistics-agnostic layer

The fermionic and bosonic lines proved the same shape of facts twice вҖ” an algebraic Fock space
built from an occupation-state type, and a basis-diagonal free-Hamiltonian evolution вҖ” using
different concrete occupation types (`FermionOccupation Mode := Finset Mode` vs. `Occupation Mode
:= Mode вҶ’вӮҖ в„•`) but genuinely identical proofs otherwise. Extracted into `Common/`:

**Design principle for what belongs in `Common/`.** `Common/` is not a dumping ground for
"whatever happened to generalize" вҖ” it is the layer both statistics enter from their different
concrete representations. A definition/theorem belongs here only if: (1) it doesn't depend on the
concrete occupation-state type; (2) `Common/` never imports `Fermionic/`/`Bosonic/` (the
dependency direction is one-way: statistics-specific code depends on `Common/`, never the
reverse); (3) in general both statistics' concrete instantiations land in the *same* PR, so
`Common/` never accumulates an API only one side actually uses; (4) if one side genuinely can't
supply an instantiation, the docstring says why. **This does not mean forcing artificial
symmetry** вҖ” the *algebraic* layer (Fock space, exchange commutator, occupation-basis interface)
is symmetric by construction, but genuinely different analytic content stays on each statistics'
own side: bosonic occupation-number sums are infinite series (even for a finite mode set, since
`Occupation Mode := Mode вҶ’вӮҖ в„•` is unbounded per mode) where fermionic ones are `Finset.sum`s
(`FermionOccupation Mode := Finset Mode` is finite), so the bosonic thermal-trace layer needs its
own convergence-aware implementation, not a forced fit into `Fermionic/ThermalExpectation.lean`'s
finite-sum shape.

## Phase 9: finite-temperature Green functions and time ordering

The genuine theorem needs imaginary-time evolution, time ordering, thermal `n`-point correlators,
Wick's theorem (BlochвҖ“de Dominicis), and the non-commutative time-ordered Dyson series. Planned
order:

1. `Fermionic/ImaginaryTimeEvolution.lean` вҖ” `e^{П„HвӮҖ}` and Heisenberg-picture evolution (**done**, see below).
2. `Fermionic/ThermalTimeOrdering.lean` вҖ” imaginary-time ordering `T_П„` (**done**, see below).
3. `Fermionic/ThermalGreenFunction.lean` вҖ” `G_{ij}(П„,П„') := -вҹЁT_П„ c_i(П„) c_jвҖ (П„')вҹ©_ОІ` (**done**, see below;
   only the free-Hamiltonian `GвӮҖ`, and not yet as a special case of a general `n`-point correlator).
4. `Fermionic/ThermalContraction.lean` вҖ” same-type contractions vanish (**done**, see below).
   `BlochDeDominicis.lean` вҖ” the general finite-mode fermionic Wick/BlochвҖ“de Dominicis theorem,
   not yet started.
5. `DysonExpansionFermionic.lean` вҖ” the genuine interaction-picture Dyson series (the name
   `Fermionic/FormalExp.lean` deliberately avoided).
6. `DiagramConnectedness.lean` вҖ” connecting Dyson-series terms to diagrams and Track B's
   connectedness result.
7. Reassemble `Common/QuantumLinkedCluster.lean` on top of the above.

**Step 1 done, in `Fermionic/ImaginaryTimeEvolution.lean`:**
- `imaginaryTimeEvolveFree Оө П„` вҖ” `e^{П„HвӮҖ}` for the free Hamiltonian, defined directly on the
  occupation-number basis via `Complex.exp` of each basis state's eigenvalue `E(n) := ОЈбөўвҲҲn Оө(i)`
  (no Taylor truncation needed, unlike `Fermionic/FormalExp.lean`, since `freeHamiltonian` is

  law and mutual inverses.
- `imaginaryTimeEvolve Оө П„ A := e^{П„HвӮҖ} A e^{-П„HвӮҖ}` вҖ” Heisenberg-picture evolution of a general
  operator, with sanity checks `imaginaryTimeEvolve_zero`/`_freeHamiltonian`.
