# Caveats

Known pitfalls, past mistakes, and things to watch out for. Add an entry whenever an error costs time so it is not repeated.

## Formalization pitfalls

- **No trace-class/Schatten-class theory in Mathlib (as of the Mathlib revision pinned in `lakefile.toml`).** `LinearMap.trace` requires finite-dimensionality. This blocks a direct extension of `QuantumTheory.DensityOperator` (`LeanCondensedMatter/QuantumTheory/DensityOperator.lean`) to the countably-infinite-dimensional lattice setting chosen in `notes/model-and-assumptions.md` for the Linked Cluster Theorem target вҖ” `Z = tr e^{-ОІH}` needs a trace on an infinite-dimensional space. Until such machinery exists (in Mathlib or built here), density-operator work stays scoped to finite-dimensional `H`.

- **Partition-lattice MГ¶bius closed-form formula `Ој(ПҖ,Пғ) = (-1)^(|ПҖ|-|Пғ|) вҲҸ(n_B-1)!` is genuinely hard to formalize вҖ” three routes tried, all blocked on missing Mathlib lemmas.** Context: `LeanCondensedMatter/Combinatorics/PartitionLattice.lean` has a `LocallyFiniteOrder (Finpartition s)` instance (done, builds cleanly), which lets Mathlib's `IncidenceAlgebra.mu`/`moebius_inversion_top`/`bot` apply to the partition lattice. But the *closed-form value* of `mu` on the partition lattice is not in Mathlib and needs its own proof. Three approaches were tried in one session, each hitting a different missing-lemma wall:
  1. **Order-isomorphism invariance of `mu` across different poset types**, i.e. a general theorem `[a,b] вүғ [a',b']` (order-iso) `вҹ№ Ој(a,b) = Ој(a',b')`, to be combined with the interval-structure isomorphism `[ПҖ,Пғ] вү… вҲҸ_{BвҲҲПғ} Finpartition(ПҖ_B)` (mirroring the standard textbook proof, e.g. Stanley EC1). Attempted directly by strong induction on `(Finset.Icc a b).card`, transporting the recursive sum via the bijection. The bookkeeping (mapping `Set.Ico`/`Finset.Ico` membership back and forth across the two posets, restricting the bijection to sub-intervals for the inductive step) got tangled and was abandoned mid-proof (not committed).
  2. **Algebraic route via inverse-uniqueness in the incidence algebra**, mirroring how Mathlib itself proves `IncidenceAlgebra.mu_prod_mu` (product-poset multiplicativity) вҖ” by showing a candidate function is `О¶`'s inverse (`f * zeta = 1`) rather than re-deriving uniqueness by hand. This avoids induction for the *product* case, but still needs an order-iso transport step to connect `[ПҖ,Пғ]` (living in `Finpartition S`) to the literal product poset `вҲҸ_B Finpartition(ПҖ_B)` (a different type) вҖ” i.e. it doesn't eliminate route 1's core difficulty, only reduces how much of the argument needs hand-rolled induction.
  3. **Direct strong induction on `|ПҖ|` (block count) entirely within `Finpartition S`**, using Mathlib's `mu_eq_neg_sum_Ioc_of_ne` (right-recursion: `Ој(ПҖ,вҠӨ) = -ОЈ_{ПҖ<ПҒвүӨвҠӨ} Ој(ПҒ,вҠӨ)`) вҖ” avoids crossing poset types at all, since strict refinement always strictly decreases block count. This reduces the goal to a *Stirling-number identity*: `ОЈ_{j=1}^{k} S(k,j)(-1)^{j-1}(j-1)! = 0` (`S` = Stirling numbers of the second kind, `Nat.stirlingSecond` in Mathlib), plus a bijective correspondence between "coarsenings of a `k`-block partition into `j` blocks" and `Nat.stirlingSecond k j`. **Neither exists in Mathlib** (checked `Mathlib/Combinatorics/Enumerative/Stirling.lean` вҖ” only basic recurrences, no cross-Stirling-kind alternating-sum identity).
  - **Conclusion:** all three routes bottom out in a genuinely-missing Mathlib lemma, not just tedious bookkeeping. This is realistically a multi-session undertaking on its own. If retried, route 3 (Stirling-number identity) is probably the most self-contained next attempt, since it stays within one poset type and reduces to a single well-defined numeric identity rather than type-crossing isomorphism transport.

- **The *abstract* cumulant formula (`c(S) = ОЈ_ПҖ mu(ПҖ,вҠӨ) вҲҸ m(B)`, via MГ¶bius inversion) does not need the closed-form MГ¶bius formula above вҖ” but still needs its own nontrivial bijection.** Realization: MГ¶bius *inversion* itself (`IncidenceAlgebra.moebius_inversion_top`/`bot`, already usable via the `LocallyFiniteOrder (Finpartition s)` instance) only needs the *abstract* `mu`, not its closed-form value вҖ” so the cumulant formula can in principle be derived now, without waiting on the closed-form-MГ¶bius problem above. What it does need is the decomposition `G(Пғ) = ОЈ_{ПҖвүӨПғ} F(ПҖ)` (`G(Пғ) := вҲҸ_{BвҲҲПғ.parts} m(B)`, `F(Пғ) := вҲҸ_{BвҲҲПғ.parts} c(B)`, given the moment-cumulant hypothesis `m(T) = ОЈ_{ПҖ:Finpartition T} вҲҸ_{BвҲҲПҖ.parts} c(B)`), which itself needs the bijection `{ПҖ // ПҖ вүӨ Пғ} вүғ О  B вҲҲ Пғ.parts, Finpartition B`.
  - **Progress made:** `Finpartition.bind_le` (`P.bind Q вүӨ P`) and `Finpartition.bind_restrict_eq_of_le` (`ПҖ вүӨ Пғ вҶ’ Пғ.bind (fun B hB => ПҖ.restrict (Пғ.le hB)) = ПҖ`, the "gluing recovers the original" direction) are proved in `LeanCondensedMatter/Combinatorics/PartitionLattice.lean`.
  - **Still needed:** (a) the *other* round-trip direction (`(Пғ.bind Q).restrict (Пғ.le hB) = Q B hB`), to get a genuine bijection rather than a one-directional map; (b) converting the "product of sums = sum over choice-functions of products" step (`вҲҸ_{BвҲҲПғ.parts} m(B) = ОЈ_{(ПҒ_B)_B} вҲҸ_B вҲҸ_{CвҲҲПҒ_B.parts} c(C)`) into Mathlib's `Finset.prod_sum`/`Finset.pi` framework, which expects each per-index choice set to be a `Finset` inside a common ambient ("`s.pi t`") type вҖ” `Finpartition B` is a `Fintype`, not literally elements of a `Finset` in a shared ambient type, so connecting the two frameworks is itself nontrivial work, not yet attempted.

- **`IsStarNormal.instContinuousFunctionalCalculus` (the base `ContinuousFunctionalCalculus в„Ӯ A IsStarNormal` instance for any `CStarAlgebra A`) is only a `local instance` in `Mathlib/Analysis/CStarAlgebra/ContinuousFunctionalCalculus/Basic.lean`, not globally registered.** Every downstream instance built on top of it (e.g. `IsSelfAdjoint.instContinuousFunctionalCalculus` in `Instances.lean`, needed to use `cfc` on `H вҶ’L[в„Ӯ] H` at all) genuinely requires `[ContinuousFunctionalCalculus в„Ӯ A IsStarNormal]` in scope вҖ” it is not auto-derived just from `[CStarAlgebra A]`. Symptom: `cfc f T` fails with `failed to synthesize instance ContinuousFunctionalCalculus в„қ (H вҶ’L[в„Ӯ] H) ?m` even though the `CStarAlgebra (H вҶ’L[в„Ӯ] H)` instance is available. **Fix:** `import Mathlib.Analysis.CStarAlgebra.ContinuousFunctionalCalculus.Basic` and add `attribute [local instance] IsStarNormal.instContinuousFunctionalCalculus` in the consuming file (see `LeanCondensedMatter/Analysis/CFC.lean`).

- **`trace`'s scalar-multiplication linearity (`trace (c вҖў T) = c * trace T`) needs a reindexing `Equiv (EigenvectorIndex T) (EigenvectorIndex (c вҖў T))` between two structurally different `Sigma` types вҖ” attempted, hit a real build wall, reverted (not committed).** Context: `LeanCondensedMatter/Analysis/CompactSelfAdjoint.lean`'s `EigenvectorIndex T := ОЈ Ој : {Ој : в„қ // Ој вү  0}, Fin (finrank в„Ӯ (eigenspace T Ој))`. The submodule-level fact `eigenspace (c вҖў T) (c * Ој) = eigenspace T Ој` (for `c вү  0`) is short and unproblematic on its own (`ext v; simp [Module.End.mem_eigenspace_iff]; ...`). The blocker is building the `Equiv` between the two *index types*: the `Fin`-valued second component of each `Sigma` pair must be transported along `Fin.cast` using `congrArg (Module.finrank в„Ӯ) (eigenspace_smul ...)`, and proving `left_inv`/`right_inv` (that composing the two `Fin.cast`s is the identity) via `ext`. In practice, building this вҖ” even just the standalone `eigenspace_smul` lemma in isolation вҖ” caused `lake build` to hang for 10+ minutes with `lean.exe` CPU time barely advancing (a few CPU-seconds over many minutes of wall-clock time), on a machine that otherwise builds this file's much harder theorems (`hasSum_eigenvectorFamily`, `orthogonal_closure_span_eigenvectorFamily`) in under 2 minutes. This looks like a genuine elaborator performance issue (likely from the nested `Fin.cast`/dependent-`Sigma`-equality proof obligations), not ordinary system load вҖ” confirmed by re-running the *unchanged* file (identical to the last successful commit) and seeing the same multi-minute slowness pattern recur, then disappear once the new code was removed.
  - **If retried:** avoid `Fin.cast` chains inside a `Sigma`-typed `Equiv`; consider instead defining the reindexing directly on `Finset.univ.val.map`/`Multiset` level (mirroring the `charpoly`-multiset technique used successfully in `LeanCondensedMatter/QuantumTheory/Entropy.lean`'s `diagOp_eigenvalues_map_eq`), or restructure `EigenvectorIndex` itself to avoid a dependent `Fin (finrank ...)` fiber (e.g. index directly by `(Ој : {Ој // Ој вү  0}) Г— eigenspace T Ој`-style existing basis vectors rather than by abstract `Fin` positions), which may sidestep the need to relate `Fin` types across two different operators altogether.
  - **Cyclicity** (`trace (S вҲҳ T) = trace (T вҲҳ S)`) is expected to be at least as hard, since it needs relating `EigenvectorIndex (S вҲҳ T)` to *both* `S`'s and `T`'s data with no simple closed-form relationship вҖ” not attempted.
