# Caveats

Known pitfalls, past mistakes, and things to watch out for. Add an entry whenever an error costs time so it is not repeated.

## Formalization pitfalls

- **No trace-class/Schatten-class theory in Mathlib (as of the Mathlib revision pinned in `lakefile.toml`).** `LinearMap.trace` requires finite-dimensionality. This blocks a direct extension of `QuantumTheory.DensityOperator` (`LeanCondensedMatter/QuantumTheory/DensityOperator.lean`) to the countably-infinite-dimensional lattice setting chosen in `notes/model-and-assumptions.md` for the Linked Cluster Theorem target — `Z = tr e^{-βH}` needs a trace on an infinite-dimensional space. Until such machinery exists (in Mathlib or built here), density-operator work stays scoped to finite-dimensional `H`.

- **Partition-lattice Möbius closed-form formula `μ(π,σ) = (-1)^(|π|-|σ|) ∏(n_B-1)!` is genuinely hard to formalize — three routes tried, all blocked on missing Mathlib lemmas.** Context: `LeanCondensedMatter/Combinatorics/PartitionLattice.lean` has a `LocallyFiniteOrder (Finpartition s)` instance (done, builds cleanly), which lets Mathlib's `IncidenceAlgebra.mu`/`moebius_inversion_top`/`bot` apply to the partition lattice. But the *closed-form value* of `mu` on the partition lattice is not in Mathlib and needs its own proof. Three approaches were tried in one session, each hitting a different missing-lemma wall:
  1. **Order-isomorphism invariance of `mu` across different poset types**, i.e. a general theorem `[a,b] ≃ [a',b']` (order-iso) `⟹ μ(a,b) = μ(a',b')`, to be combined with the interval-structure isomorphism `[π,σ] ≅ ∏_{B∈σ} Finpartition(π_B)` (mirroring the standard textbook proof, e.g. Stanley EC1). Attempted directly by strong induction on `(Finset.Icc a b).card`, transporting the recursive sum via the bijection. The bookkeeping (mapping `Set.Ico`/`Finset.Ico` membership back and forth across the two posets, restricting the bijection to sub-intervals for the inductive step) got tangled and was abandoned mid-proof (not committed).
  2. **Algebraic route via inverse-uniqueness in the incidence algebra**, mirroring how Mathlib itself proves `IncidenceAlgebra.mu_prod_mu` (product-poset multiplicativity) — by showing a candidate function is `ζ`'s inverse (`f * zeta = 1`) rather than re-deriving uniqueness by hand. This avoids induction for the *product* case, but still needs an order-iso transport step to connect `[π,σ]` (living in `Finpartition S`) to the literal product poset `∏_B Finpartition(π_B)` (a different type) — i.e. it doesn't eliminate route 1's core difficulty, only reduces how much of the argument needs hand-rolled induction.
  3. **Direct strong induction on `|π|` (block count) entirely within `Finpartition S`**, using Mathlib's `mu_eq_neg_sum_Ioc_of_ne` (right-recursion: `μ(π,⊤) = -Σ_{π<ρ≤⊤} μ(ρ,⊤)`) — avoids crossing poset types at all, since strict refinement always strictly decreases block count. This reduces the goal to a *Stirling-number identity*: `Σ_{j=1}^{k} S(k,j)(-1)^{j-1}(j-1)! = 0` (`S` = Stirling numbers of the second kind, `Nat.stirlingSecond` in Mathlib), plus a bijective correspondence between "coarsenings of a `k`-block partition into `j` blocks" and `Nat.stirlingSecond k j`. **Neither exists in Mathlib** (checked `Mathlib/Combinatorics/Enumerative/Stirling.lean` — only basic recurrences, no cross-Stirling-kind alternating-sum identity).
  - **Conclusion:** all three routes bottom out in a genuinely-missing Mathlib lemma, not just tedious bookkeeping. This is realistically a multi-session undertaking on its own. If retried, route 3 (Stirling-number identity) is probably the most self-contained next attempt, since it stays within one poset type and reduces to a single well-defined numeric identity rather than type-crossing isomorphism transport.

- **The *abstract* cumulant formula (`c(S) = Σ_π mu(π,⊤) ∏ m(B)`, via Möbius inversion) does not need the closed-form Möbius formula above — but still needs its own nontrivial bijection.** Realization: Möbius *inversion* itself (`IncidenceAlgebra.moebius_inversion_top`/`bot`, already usable via the `LocallyFiniteOrder (Finpartition s)` instance) only needs the *abstract* `mu`, not its closed-form value — so the cumulant formula can in principle be derived now, without waiting on the closed-form-Möbius problem above. What it does need is the decomposition `G(σ) = Σ_{π≤σ} F(π)` (`G(σ) := ∏_{B∈σ.parts} m(B)`, `F(σ) := ∏_{B∈σ.parts} c(B)`, given the moment-cumulant hypothesis `m(T) = Σ_{π:Finpartition T} ∏_{B∈π.parts} c(B)`), which itself needs the bijection `{π // π ≤ σ} ≃ Π B ∈ σ.parts, Finpartition B`.
  - **Progress made:** `Finpartition.bind_le` (`P.bind Q ≤ P`) and `Finpartition.bind_restrict_eq_of_le` (`π ≤ σ → σ.bind (fun B hB => π.restrict (σ.le hB)) = π`, the "gluing recovers the original" direction) are proved in `LeanCondensedMatter/Combinatorics/PartitionLattice.lean`.
  - **Still needed:** (a) the *other* round-trip direction (`(σ.bind Q).restrict (σ.le hB) = Q B hB`), to get a genuine bijection rather than a one-directional map; (b) converting the "product of sums = sum over choice-functions of products" step (`∏_{B∈σ.parts} m(B) = Σ_{(ρ_B)_B} ∏_B ∏_{C∈ρ_B.parts} c(C)`) into Mathlib's `Finset.prod_sum`/`Finset.pi` framework, which expects each per-index choice set to be a `Finset` inside a common ambient ("`s.pi t`") type — `Finpartition B` is a `Fintype`, not literally elements of a `Finset` in a shared ambient type, so connecting the two frameworks is itself nontrivial work, not yet attempted.

- **`IsStarNormal.instContinuousFunctionalCalculus` (the base `ContinuousFunctionalCalculus ℂ A IsStarNormal` instance for any `CStarAlgebra A`) is only a `local instance` in `Mathlib/Analysis/CStarAlgebra/ContinuousFunctionalCalculus/Basic.lean`, not globally registered.** Every downstream instance built on top of it (e.g. `IsSelfAdjoint.instContinuousFunctionalCalculus` in `Instances.lean`, needed to use `cfc` on `H →L[ℂ] H` at all) genuinely requires `[ContinuousFunctionalCalculus ℂ A IsStarNormal]` in scope — it is not auto-derived just from `[CStarAlgebra A]`. Symptom: `cfc f T` fails with `failed to synthesize instance ContinuousFunctionalCalculus ℝ (H →L[ℂ] H) ?m` even though the `CStarAlgebra (H →L[ℂ] H)` instance is available. **Fix:** `import Mathlib.Analysis.CStarAlgebra.ContinuousFunctionalCalculus.Basic` and add `attribute [local instance] IsStarNormal.instContinuousFunctionalCalculus` in the consuming file (see `LeanCondensedMatter/Analysis/CFC.lean`).

- **`trace`'s scalar-multiplication linearity (`trace (c • T) = c * trace T`) is now proved (`ContinuousLinearMap.trace_smul` in `LeanCondensedMatter/Analysis/CompactSelfAdjoint.lean`) — but only after several genuinely-expensive dead ends building a reindexing `Equiv (EigenvectorIndex T) (EigenvectorIndex (c • T))` between the two `Sigma` types.** Context: `EigenvectorIndex T := Σ μ : {μ : ℝ // μ ≠ 0}, Fin (finrank ℂ (eigenspace T μ))`. The submodule-level fact `eigenspace (c • T) (c * μ) = eigenspace T μ` (for `c ≠ 0`) is short and unproblematic (`ext v; simp [Module.End.mem_eigenspace_iff]; ...`). What genuinely does not work is building an `Equiv` directly between the two *dependent Sigma index types* and then extracting facts about it:
  - Building the equiv via `Equiv.sigmaCongr` (composing `sigmaCongrRight`/`sigmaCongrLeft`/`Equiv.trans`) type-checked, but any later attempt to compute `(that equiv a).fst`/`.snd` — whether via `rfl`, `dsimp`+`rw`, or `simp` with the relevant `_apply` lemmas — either hit `DFunLike`/`EquivLike` instance mismatches (because `EigenvectorIndex T` is a semireducible `def` wrapping `Sigma`, and tactics partially unfold it inconsistently) or, when those were worked around, hit a genuine **`(kernel) deterministic timeout`** — confirmed reproducible (not just slow, actually failing) on wall-clock times up to ~1200s of real `lean.exe` CPU time.
  - Building the equiv *directly* by hand (`Sigma.mk`-valued `toFun`/`invFun`, `Sigma.ext`/`Fin.heq_ext_iff` for `left_inv`/`right_inv`) fixed the `.fst`-projection cost (a plain `Sigma.mk` projection is cheap), but the `HEq` proof obligations for `left_inv`/`right_inv` — even when reduced to individually shallow, separately-`rfl`-provable lemmas about single `finCongr` applications — *also* hit kernel timeouts once their statements involved `Module.finrank ℂ (Module.End.eigenspace ...)` in the type. The cost is not about proof-term complexity (those goals are `Prop`s, hence proof-irrelevant) but about the kernel re-verifying that the *stated types* on both sides of an `Eq`/`rfl` are defeq, which forces unfolding `Module.finrank`/`Module.End.eigenspace`/operator coercions regardless of tactic choice.
  - **What actually worked:** avoid ever building or reindexing through `Equiv (EigenvectorIndex T) (EigenvectorIndex (c • T))` at all. Instead split each trace-defining `tsum` into an *outer* `tsum` over the non-dependent base type `{μ : ℝ // μ ≠ 0}` and an *inner, finite* `Finset.sum`/`tsum_fintype` over `Fin (finrank ...)`, using `Summable.tsum_sigma`. Since the summand `a ↦ a.1.1` doesn't depend on the `Fin` component, the inner sum is just `finrank • μ.1` (`Finset.sum_const` + `Fintype.card_fin`) — a plain `ℕ`-level fact from `eigenspace_smul`, no `Fin.cast`/`HEq` needed anywhere. The outer reindexing only needs an ordinary (non-dependent) `Equiv` on `{μ : ℝ // μ ≠ 0}` (`eigenvalueScaleEquiv`), which is cheap. See `tsum_eigenvectorIndex_eq_tsum_mul_finrank` for the reusable split.
  - **Lesson for future dependent-`Sigma`-reindexing needs:** if the fiber type only affects a *summand that's independent of the fiber index* (as here), always prefer splitting the sum (`tsum_sigma`) over reindexing the `Sigma` type itself — it sidesteps `Fin.cast`/`HEq` entirely. Reindexing the dependent type directly should be treated as a last resort even when it type-checks, since downstream lemmas about it can still hit real (not just slow) kernel timeouts.
  - **Cyclicity** (`trace (S ∘ T) = trace (T ∘ S)`) is expected to be harder still, since it needs relating `EigenvectorIndex (S ∘ T)` to *both* `S`'s and `T`'s data with no simple closed-form relationship, and the "split the sum, don't reindex the type" trick above may not directly apply — not attempted.
